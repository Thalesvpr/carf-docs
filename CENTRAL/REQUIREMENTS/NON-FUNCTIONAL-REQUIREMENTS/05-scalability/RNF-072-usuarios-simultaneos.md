# RNF-072: Usuários Simultâneos

O sistema composto pelos módulos GEOAPI e GEOWEB deve ser capaz de suportar pelo menos 500 usuários simultâneos realizando operações típicas sem que ocorra degradação significativa de performance ou aumento excessivo de taxas de erro, garantindo que o sistema possa atender cenários de uso intenso como períodos de pico de trabalho, eventos especiais ou crescimento orgânico da base de usuários. A capacidade de atender 500 usuários concorrentes deve ser medida através de testes de carga (load tests) que utilizam ferramentas especializadas como k6, Artillery, JMeter ou Locust para simular 500 usuários virtuais executando fluxos realistas de interação com o sistema, incluindo ações como login, navegação entre páginas, consultas de dados, visualização de mapas, criação e edição de registros, upload de arquivos e outras operações representativas do uso real do sistema. Durante esses testes de carga, o tempo de resposta médio e percentilado (p95) não deve exceder o dobro dos valores baseline medidos sob carga leve, garantindo que mesmo sob carga pesada os usuários ainda experimentem performance aceitável, reconhecendo que alguma degradação é esperada mas deve ser mantida dentro de limites razoáveis que não prejudiquem significativamente a experiência do usuário. A taxa de erro (error rate) durante os testes de carga não deve exceder 1%, garantindo que a grande maioria das requisições seja processada com sucesso mesmo sob carga intensa, onde erros podem incluir timeouts, falhas de conexão, erros 500 do servidor ou qualquer outra condição que impeça o usuário de completar a operação desejada. Os testes de carga devem ser executados regularmente, idealmente como parte do pipeline de CI/CD antes de releases importantes, e os resultados devem ser comparados com baselines anteriores para identificar regressões de performance introduzidas por mudanças no código ou configuração. O sistema deve incluir mecanismos de proteção contra sobrecarga como rate limiting, circuit breakers e graceful degradation que previnem que o sistema se torne completamente indisponível sob carga extrema, garantindo que mesmo quando a capacidade é excedida, o sistema continue atendendo pelo menos uma porção dos usuários ao invés de falhar completamente para todos. Suportar 500 usuários simultâneos demonstra que o sistema foi projetado com escalabilidade em mente, fornece margem de segurança para crescimento futuro sem necessidade de refatorações arquiteturais significativas, e garante que o sistema possa atender organizações de porte médio que podem ter várias dezenas ou centenas de funcionários utilizando o sistema simultaneamente durante horários de pico. Este requisito é classificado como should-have, sendo altamente desejável para garantir que o sistema seja viável para implantações de escala realista, demonstrando que considerações de performance e escalabilidade foram incorporadas desde o design inicial, e fornecendo confiança aos stakeholders de que o investimento no sistema não será limitado por restrições técnicas quando a adoção crescer.
