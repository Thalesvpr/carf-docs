# RNF-047: Internacionalização

O sistema deve suportar Português Brasileiro (PT-BR) como idioma obrigatório e implementar infraestrutura de internacionalização (i18n) que permita futura adição de outros idiomas como inglês e espanhol sem necessidade de refatoração significativa do código. A externalização de textos deve ser implementada utilizando biblioteca i18next ou similar no frontend React/React Native, onde todas as strings visíveis ao usuário são removidas do código e substituídas por chaves de tradução, por exemplo ao invés de `<button>Salvar</button>` usar `<button>{t('common.save')}</button>`, com arquivos JSON organizados por namespace e idioma como `pt-BR/common.json`, `pt-BR/forms.json`, `pt-BR/errors.json` contendo as traduções correspondentes, facilitando manutenção e permitindo que tradutores trabalhem diretamente nos arquivos JSON sem necessidade de conhecimento técnico do código. O formato de data e hora deve seguir padrões brasileiros utilizando bibliotecas como date-fns ou luxon com locale pt-BR configurado, exibindo datas no formato DD/MM/YYYY ao invés do formato americano MM/DD/YYYY, horários no formato 24 horas como 14:30 ou opcionalmente 12 horas com AM/PM conforme preferência do usuário, nomes de meses e dias da semana em português (Janeiro, Fevereiro, Segunda-feira, Terça-feira), e fusos horários relevantes ao Brasil incluindo BRT (Brasília Time UTC-3) e BRST (Brasília Summer Time UTC-2) embora dados no backend sejam sempre armazenados em UTC conforme RNF-089. A formatação de moeda deve utilizar padrão brasileiro BRL (Real) com símbolo R$ posicionado antes do valor, separador de milhares como ponto (.) e separador decimal como vírgula (,), exibindo valores como R$ 1.234,56 através de bibliotecas de formatação que respeitam locale pt-BR como Intl.NumberFormat em JavaScript com parâmetros { style: 'currency', currency: 'BRL', locale: 'pt-BR' }, aplicado consistentemente em toda a aplicação incluindo relatórios, dashboards e exportações. A estrutura para adicionar novos idiomas deve ser estabelecida desde o início, criando estrutura de pastas que acomode múltiplos locales como `locales/pt-BR/`, `locales/en-US/`, `locales/es-ES/`, implementando seletor de idioma na interface mesmo que inicialmente mostre apenas PT-BR, armazenando preferência de idioma do usuário no perfil ou localStorage, e garantindo que componentes reagem dinamicamente a mudanças de locale sem necessidade de recarregar página, usando recursos como i18next.changeLanguage() e rerender automático via hooks ou context. As considerações especiais para aplicativo mobile REURBCAD incluem suporte a idioma do sistema operacional detectado automaticamente via APIs do React Native, fallback para PT-BR se idioma do sistema não for suportado, e sincronização de preferência de idioma entre múltiplos dispositivos do mesmo usuário através do backend. Os critérios de aceitação incluem a verificação de que todos os textos visíveis ao usuário são externalizados usando i18next com arquivos de tradução organizados por namespace, que datas e horas são formatadas no padrão brasileiro (DD/MM/YYYY, 24h) usando bibliotecas com locale pt-BR, que valores monetários são exibidos como R$ X.XXX,XX em toda a aplicação usando formatação consistente, que existe estrutura de pastas e configuração para suportar múltiplos idiomas (en-US, es-ES) mesmo que traduções ainda não estejam completas, e que mudança de idioma (quando implementada) atualiza interface em tempo real sem necessidade de reload. Este requisito é should-have por adicionar valor futuro permitindo expansão internacional do sistema e demonstrando boas práticas de desenvolvimento, embora no contexto atual focado no Brasil, apenas PT-BR seja estritamente necessário, com a infraestrutura de i18n sendo mais um investimento para facilitar evolução futura do que necessidade imediata, podendo ser priorizado após funcionalidades core estarem estáveis.
