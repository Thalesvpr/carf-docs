# RNF-055: Atalhos de Teclado

A aplicação GEOWEB deve implementar atalhos de teclado para ações frequentes que permitem usuários avançados operarem mais eficientemente sem necessidade de navegar menus ou clicar em botões, seguindo convenções estabelecidas de aplicações desktop e web modernas para minimizar curva de aprendizado. O atalho Ctrl+S para salvar deve ser implementado globalmente em todas as telas que possuem formulários de edição ou criação de dados, interceptando o evento de teclado padrão que normalmente tentaria salvar a página HTML e prevenindo este comportamento através de event.preventDefault(), disparando em vez disso a lógica de salvamento da aplicação que valida dados localmente, envia requisição ao backend via API e mostra feedback de sucesso ou erro, funcionando tanto em formulários de unidades habitacionais, titulares, comunidades quanto em outros contextos de edição, com indicação visual sutil como tooltip ou hint text mostrando "Ctrl+S para salvar" próximo a botão de save para educar usuários sobre disponibilidade do atalho. O atalho Ctrl+F para buscar deve ativar campo de busca global da aplicação, movendo foco automaticamente para input de busca e selecionando qualquer texto existente para facilitar nova busca, sobrescrevendo comportamento padrão do navegador de busca na página quando apropriado ou complementando com busca in-app mais poderosa que procura em dados estruturados ao invés de apenas texto renderizado, útil para rapidamente encontrar registros sem necessidade de navegar até página de busca ou clicar em campo de busca. O atalho Esc para fechar modais deve implementar padrão universal onde pressionar tecla Escape fecha o modal, dialog ou panel overlay atualmente aberto e retorna foco para elemento que ativou o modal, funcionando em confirmações de exclusão, formulários em modal, visualizadores de imagem fullscreen, painéis laterais de filtros e qualquer outro overlay, com possível confirmação antes de fechar se modal contém dados não salvos através de diálogo "Tem certeza que deseja sair sem salvar?". A listagem de atalhos acessível via Ctrl+? ou Ctrl+/ deve abrir modal ou painel lateral mostrando referência completa de todos os atalhos disponíveis no contexto atual, organizados por categoria (Navegação, Edição, Visualização, etc.), mostrando combinação de teclas de forma visual clara como caixas com `Ctrl` + `S` ao invés de texto plano "Ctrl+S", indicando atalhos globais que funcionam em toda aplicação versus atalhos contextuais específicos a determinada tela ou modo, e servindo como documentação auto-descritiva que usuários podem consultar a qualquer momento sem buscar em documentação externa. Os atalhos adicionais úteis podem incluir Ctrl+N para criar novo registro no contexto atual, Ctrl+E para entrar em modo de edição, Tab e Shift+Tab para navegar entre campos de formulário com ordem lógica, Ctrl+K para comando palette ou quick actions similar a Spotlight/Command Palette em IDEs modernos, setas direcionais para navegar em listas e mapas, e atalhos numéricos como 1-9 para trocar entre tabs ou seções quando aplicável, todos documentados na listagem de atalhos. A implementação deve considerar compatibilidade cross-platform adaptando para Cmd ao invés de Ctrl em macOS quando detectado sistema operacional do usuário, evitando conflitos com atalhos do navegador escolhendo combinações que não sobrescrevam funções críticas do browser, tratando edge cases como múltiplos modais abertos onde Esc fecha apenas o mais recente, e garantindo que atalhos não disparem quando usuário está digitando em campo de texto (com exceção de Esc que deve permitir sair de input). Os critérios de aceitação incluem a implementação de Ctrl+S salvando formulário atual em todas as telas de edição com feedback visual de sucesso ou erro, a ativação de campo de busca via Ctrl+F com foco automático e seleção de texto existente, o fechamento de modais e overlays via tecla Esc funcionando consistentemente em toda aplicação, a exibição de listagem completa de atalhos via Ctrl+? em modal bem formatado e organizado, e testes em Windows e macOS confirmando que atalhos funcionam corretamente adaptando Ctrl/Cmd conforme plataforma. Este requisito é could-have por adicionar valor significativo para usuários avançados que usam sistema frequentemente e se beneficiam de eficiência adicional, mas não sendo essencial para usabilidade básica especialmente considerando que muitos usuários podem preferir interação via mouse/touch e nunca descobrir ou usar atalhos de teclado, sendo razoável implementar após funcionalidades core estarem estáveis e quando investimento em power user features traz retorno em produtividade para usuários frequentes do sistema.
