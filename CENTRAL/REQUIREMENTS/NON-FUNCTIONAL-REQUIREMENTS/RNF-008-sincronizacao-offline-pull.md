# RNF-008: Sincronização Offline - Pull

O download inicial de dados de uma comunidade contendo até 5000 unidades habitacionais deve ser concluído em até 30 segundos, permitindo que usuários do aplicativo mobile REURBCAD possam rapidamente sincronizar dados para trabalho offline em campo, onde conectividade de internet pode ser intermitente ou inexistente. Este requisito aplica-se aos módulos REURBCAD e GEOAPI, onde o aplicativo mobile solicita dados através da API e armazena localmente para acesso offline. A métrica de aceitação estabelece que o tempo de download deve ser menor ou igual a 30 segundos para conjuntos de dados contendo até 5000 unidades habitacionais, incluindo suas geometrias, atributos, possuidores associados e demais dados relacionados necessários para operação offline. Os critérios de aceitação incluem a implementação obrigatória de compressão de payload utilizando gzip ou similar, reduzindo significativamente o volume de dados transferidos pela rede ao aproveitar a natureza textual e repetitiva dos dados JSON, a presença de uma barra de progresso com estimativa de tempo restante que mantém o usuário informado sobre o andamento do download e evita a percepção de travamento durante downloads mais demorados em conexões lentas, e o armazenamento dos dados em banco de dados SQLite local no dispositivo móvel, permitindo consultas eficientes e persistência dos dados mesmo quando o aplicativo é fechado ou o dispositivo reiniciado. Este requisito é classificado como Must-have porque a funcionalidade de trabalho offline é essencial para o caso de uso principal do REURBCAD, onde equipes de campo realizam cadastramento e atualização de dados em áreas que frequentemente carecem de infraestrutura de conectividade adequada. A implementação deve considerar otimização da estrutura de dados retornada pela API para incluir apenas campos essenciais ao trabalho offline, implementação de download incremental onde apenas dados novos ou modificados são transferidos em sincronizações subsequentes reduzindo drasticamente o volume de dados, uso de formatos binários ou estruturas de dados mais compactas quando apropriado ao invés de JSON puro, implementação de retomada de download interrompido para evitar necessidade de reiniciar do zero em caso de falha de conexão, e priorização de dados críticos no início do download permitindo que usuários comecem a trabalhar com dados parciais enquanto o restante continua sendo transferido em background.
