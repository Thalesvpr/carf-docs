# RNF-048: Acessibilidade (WCAG 2.1)

A aplicação GEOWEB deve implementar nível AA de acessibilidade conforme Web Content Accessibility Guidelines (WCAG) 2.1, garantindo que o sistema seja utilizável por pessoas com diferentes tipos de deficiências incluindo visual, auditiva, motora e cognitiva, promovendo inclusão digital e cumprindo potenciais requisitos legais de acessibilidade para sistemas de governo. A navegação por teclado funcional deve permitir que usuários operem toda a interface sem necessidade de mouse, implementando focus management adequado onde todos os elementos interativos (botões, links, inputs, dropdowns, modais) são acessíveis via tecla Tab em ordem lógica que reflete a estrutura visual da página, com indicadores visuais claros de qual elemento está focado através de outline ou estilos customizados que não sejam removidos via CSS, suporte a atalhos de teclado padrão como Enter para ativar botões, Escape para fechar modais, setas para navegar em listas e menus, e skip links no início da página permitindo pular direto ao conteúdo principal sem precisar tabular por toda navegação. A compatibilidade com screen reader deve ser assegurada através de semântica HTML adequada usando elementos nativos como `<button>`, `<nav>`, `<main>`, `<article>` ao invés de divs genéricas com onClick, adicionando atributos ARIA onde elementos customizados são inevitáveis incluindo aria-label para elementos sem texto visível, aria-labelledby e aria-describedby para associar labels e descrições, aria-live para anunciar mudanças dinâmicas de conteúdo, aria-expanded em elementos colapsáveis, aria-selected em tabs e listas, e role quando semântica nativa não é suficiente, testado com screen readers populares como NVDA (Windows), JAWS (Windows) e VoiceOver (macOS/iOS) para garantir que informações e funcionalidades sejam corretamente comunicadas. O contraste de cores deve atender taxa mínima de 4.5:1 para textos normais e 3:1 para textos grandes (18pt ou 14pt bold) conforme WCAG AA, validado através de ferramentas automatizadas como axe DevTools ou WAVE, garantindo que textos sejam legíveis mesmo para pessoas com baixa visão ou daltonismo, escolhendo paleta de cores cuidadosamente para que informações não sejam comunicadas apenas por cor (por exemplo em gráficos ou status), adicionando sempre indicadores textuais ou patterns visuais complementares. As labels em formulários devem ser explícitas e associadas corretamente aos inputs usando elemento `<label>` com atributo for correspondendo ao id do input, garantindo que ao clicar na label o input seja focado e que screen readers anunciem corretamente o propósito de cada campo, incluindo placeholders apenas como exemplo adicional nunca como substituto de label, fornecendo mensagens de erro claras associadas via aria-describedby, indicando campos obrigatórios visualmente e via aria-required, e agrupando campos relacionados em `<fieldset>` com `<legend>` descritivo. As considerações adicionais incluem fornecimento de alternativas textuais para conteúdo não-textual através de alt text em imagens, transcrições ou captions para vídeos se aplicável, descrições ARIA para ícones e gráficos complexos, tempo suficiente para usuários interagirem sem timeouts agressivos ou com opção de estender tempo, prevenção de conteúdo que pisca mais de 3 vezes por segundo para evitar gatilhos de epilepsia, e estrutura de headings hierárquica (h1, h2, h3) que facilita navegação via screen reader. Os critérios de aceitação incluem a navegação completa da aplicação usando apenas teclado sem bloqueios ou focus traps indesejados, a compatibilidade testada com pelo menos um screen reader (NVDA ou VoiceOver) confirmando que todas as funcionalidades são anunciáveis e operáveis, a verificação automatizada via axe DevTools ou Lighthouse confirmando que contraste de cores atende mínimo de 4.5:1 para textos normais, a validação de que todos os inputs de formulário têm labels explícitas corretamente associadas e campos obrigatórios são indicados, e a pontuação de acessibilidade no Lighthouse de pelo menos 90 de 100 pontos. Este requisito é should-have por promover inclusão e potencialmente ser exigência legal para sistemas governamentais, além de melhorar usabilidade geral mesmo para usuários sem deficiências, embora implementação completa de acessibilidade requeira esforço contínuo e testes especializados que podem competir com desenvolvimento de features funcionais, sendo razoável implementar incrementalmente priorizando as melhorias de maior impacto primeiro.
