# RNF-052: Consistência de UI

A interface do usuário deve seguir um design system consistente e bem documentado que estabelece padrões visuais e de interação aplicados uniformemente em todos os módulos GEOWEB e REURBCAD, criando experiência coesa que reduz carga cognitiva dos usuários ao eliminar variações arbitrárias de estilo e comportamento entre diferentes partes da aplicação. A paleta de cores padronizada deve definir cores primárias usadas para elementos principais de navegação, botões de ação principal e branding visual, cores secundárias para elementos de suporte e variações, cores semânticas para comunicar estados incluindo sucesso (verde), erro (vermelho), warning (amarelo/laranja), informação (azul), cores de fundo para diferentes níveis de hierarquia (background, surface, elevated), e cores de texto com variações para texto primário (high emphasis), secundário (medium emphasis) e desabilitado (low emphasis), todas documentadas com códigos hexadecimais específicos e nomes semânticos que facilitam referência como `color.primary.main`, `color.error.dark`, garantindo acessibilidade através de verificação de contraste adequado conforme WCAG. Os componentes reutilizáveis devem ser desenvolvidos como biblioteca centralizada, criando componentes React genéricos para elementos comuns como Button com variantes (primary, secondary, text, outlined), Input com suporte a validação e estados de erro, Select e Dropdown com comportamento consistente de abertura e seleção, Modal e Dialog com animações e posicionamento padronizados, Card para agrupamento de conteúdo, Table com paginação e ordenação, Toast para notificações temporárias, e componentes específicos do domínio como MapViewer, UnitCard, HolderForm, todos versionados e documentados com exemplos de uso e props aceitas, preferencialmente usando Storybook ou similar para visualização e teste isolado de componentes. Os espaçamentos e tipografia consistentes estabelecem ritmo visual harmonioso, definindo escala de espaçamento baseada em múltiplos de valor base (tipicamente 4px ou 8px) como 4px, 8px, 16px, 24px, 32px, 48px, 64px usados para margins, paddings e gaps, e escala tipográfica com tamanhos, pesos e line-heights definidos para diferentes níveis de hierarquia como Heading 1 (32px, bold, 1.2 line-height), Heading 2 (24px, semi-bold, 1.3), Body (16px, regular, 1.5), Caption (14px, regular, 1.4), utilizando famílias de fontes consistentes preferencialmente web-safe ou carregadas via Google Fonts/Adobe Fonts com fallbacks apropriados. As guidelines de interação devem documentar comportamentos esperados como feedback visual ao hover (mudança sutil de cor ou elevação), estados de foco para navegação por teclado (outline visível), estados de loading (spinners, skeleton screens), transições e animações (duração, easing functions), comportamento de modais (como abrem, como fecham, se permitem fechar clicando fora), validação de formulários (quando validar, como mostrar erros), e padrões de navegação (breadcrumbs, tabs, navegação lateral), garantindo que mesmos tipos de interação funcionem da mesma forma em toda aplicação. A documentação do design system deve ser mantida atualizada e acessível a todos os membros da equipe, incluindo guidelines visuais com exemplos de uso correto e incorreto, código de componentes com API documentada, princípios de design que guiam decisões, e processo de contribuição para adicionar ou modificar componentes, utilizando ferramentas como Figma para designs de alta fidelidade, Storybook para documentação interativa de componentes, e arquivo README ou wiki para guidelines textuais. Os critérios de aceitação incluem a definição e documentação de paleta de cores completa com pelo menos cores primária, secundária, semânticas (sucesso, erro, warning, info) e suas variações, a criação de biblioteca de componentes reutilizáveis contendo pelo menos 15 componentes comuns (Button, Input, Modal, Card, etc.) usados em pelo menos 80% das telas, a aplicação consistente de escala de espaçamento baseada em múltiplos de 8px e escala tipográfica com 4-5 níveis de heading mais body e caption, a verificação em code review de que novos desenvolvimentos utilizam componentes existentes ao invés de criar variações, e a existência de documentação acessível do design system via Storybook ou similar com exemplos de todos os componentes. Este requisito é should-have por melhorar significativamente a qualidade percebida e usabilidade do sistema através de consistência visual e de interação, reduzir tempo de desenvolvimento ao promover reuso de componentes, e facilitar manutenção ao centralizar decisões de design, embora estabelecer design system completo requeira investimento inicial considerável de tempo de design e desenvolvimento que pode competir com entrega de features funcionais, sendo razoável implementar incrementalmente começando com componentes mais usados e expandindo cobertura ao longo do tempo.
