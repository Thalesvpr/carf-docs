# RNF-031: Timeout de Sessão

As aplicações GEOWEB e REURBCAD devem implementar timeout automático de sessões inativas para prevenir acesso não autorizado quando usuários deixam dispositivos desacompanhados com sessões autenticadas abertas, através de monitoramento de atividade do usuário e invalidação de tokens após período de inatividade. O sistema deve detectar inatividade do usuário através de ausência de interações como cliques, toques na tela, digitação ou movimentos de mouse por período de 30 minutos, momento em que a sessão deve ser considerada expirada e o usuário deve ser desconectado automaticamente. Um aviso de expiração iminente deve ser apresentado ao usuário 2 minutos antes do timeout (aos 28 minutos de inatividade), através de modal ou notificação não-intrusiva que ofereça opção de "Continuar conectado" que renova a sessão se o usuário ainda estiver presente, evitando desconexões frustrantes durante pausas breves onde o usuário está próximo mas temporariamente sem interagir. Quando o timeout efetivo ocorre aos 30 minutos sem interação, o cliente deve realizar logout local limpando tokens de acesso e refresh do armazenamento, e opcionalmente notificar o servidor para revogação imediata dos tokens embora a expiração natural dos tokens de 15 minutos para access token já forneça proteção razoável. Após logout automático por timeout, o usuário deve ser redirecionado para tela de login com mensagem clara explicando que a sessão expirou por inatividade, mantendo contexto quando possível para que após novo login o usuário retorne à mesma tela onde estava trabalhando. A detecção de atividade deve considerar interações em qualquer parte da aplicação single-page, incluindo cliques em mapas, abertura de modais, edição de formulários e scrolling de listas, resetando o contador de inatividade em cada interação genuína. Para aplicação mobile REURBCAD, eventos de tela bloqueada do dispositivo devem também acionar timeout imediato ou acelerado, considerando que dispositivo bloqueado frequentemente indica que o usuário se afastou. A configuração do tempo de timeout deve ser externalizável através de variáveis de ambiente ou configuração administrativa, permitindo que diferentes organizações ajustem o período conforme suas políticas de segurança, desde timeouts mais agressivos de 10-15 minutos para ambientes de alta segurança até timeouts relaxados de 60 minutos para ambientes de menor risco. Logs de timeout devem ser registrados para análise de padrões de uso e otimização do período configurado baseado em comportamento real dos usuários. Este requisito é classificado como Should-have pois, embora timeouts de sessão sejam importante medida de segurança especialmente em terminais compartilhados ou ambientes públicos, a expiração natural de tokens já oferece proteção significativa, tornando timeout de inatividade uma camada adicional de defesa recomendada mas não absolutamente crítica para lançamento inicial.
